## Steps

- [x] **Step 1 — Project scaffold**: Created `tools/appcontainer-harness/appcontainer-harness.csproj` (`net10.0`, `OutputType=Exe`, nullable enabled), folder structure (`Profiles/`, `Launching/`, `Scenarios/`, `Reporting/`, `Models/`, `Program.cs`). Added `tools/appcontainer-harness/bin/` and `.../obj/` to `.gitignore`.
- [x] **Step 2 — CsWin32 setup**: Added `Microsoft.Windows.CsWin32` package reference and `NativeMethods.txt`. Final API list (expanded from the plan during implementation): `CreateAppContainerProfile`, `DeleteAppContainerProfile`, `DeriveAppContainerSidFromAppContainerName`, `CreateProcessW`, `InitializeProcThreadAttributeList`, `UpdateProcThreadAttribute`, `DeleteProcThreadAttributeList`, `GetExitCodeProcess`, `TerminateProcess`, `WaitForSingleObject`, `GetTokenInformation`, `OpenProcessToken`, `CloseHandle`, `FreeSid`, `ConvertSidToStringSidW`, `LocalFree`, `CreateFile`, plus explicit type requests `STARTUPINFOEXW` and `SECURITY_CAPABILITIES` (CsWin32 only generates struct types when directly requested or pulled in by a requested method's own signature - these two weren't auto-pulled in, so needed explicit listing).
- [x] **Step 3 — Domain model types**: Implemented `ExpectMode`, `ExpectedOutcome`, `Scenario`, `LaunchResult`, `ScenarioResult`, `SuiteReport` in `Models/DomainModels.cs`.
- [x] **Step 4 — Profile Manager**: Implemented in `Profiles/ProfileManager.cs`. Find-or-create via `CreateAppContainerProfile` first, falling back to `DeriveAppContainerSidFromAppContainerName` on `HRESULT 0x800700B7` (already exists) - `DeriveAppContainerSidFromAppContainerName` alone can't serve as an existence check since it computes a SID algorithmically regardless of registration. ACL grants via `icacls.exe` subprocess (read+execute on `bin`, modify on `data`). `DeleteProfile()` implements `--cleanup`.
- [x] **Step 5 — Sandboxed Process Launcher**: Implemented in `Launching/ProcessLauncher.cs`. Stages `bash.exe` + `msys-2.0.dll` into the fixed bin folder; output capture uses inheritable-handle temp files (`CreateFile` with `bInheritHandle=true`) rather than pipes, read back after the process exits - simpler and avoids pipe-buffer-deadlock complexity for a case where we don't need streaming output. 15s timeout via `WaitForSingleObject` + `TerminateProcess` on timeout.
- [x] **Step 6 — Scenario Library**: Implemented in `Scenarios/ScenarioLibrary.cs` - all 7 scenarios per the design, `file-io` scenario uses `/e/Temp/appcontainer-harness/data/` (MSYS path form of the granted `E:\Temp\...` folder).
- [x] **Step 7 — Scenario Runner/Reporter**: Implemented in `Reporting/ScenarioRunner.cs` (BR-1/BR-3/BR-4/BR-5 logic) and `Reporting/Reporter.cs` (console summary + `last-report.json` via `System.Text.Json`).
- [x] **Step 8 — CLI Entry Point**: Implemented in `Program.cs` - `--target`, `--scenario`, `--expect`, `--verbose`, `--cleanup`; repo root auto-discovered by walking up from the executable's directory looking for `aidlc-docs/`.
- [x] **Step 9 — Build**: `dotnet build` succeeded (0 warnings, 0 errors) after resolving CsWin32 friendly-vs-raw P/Invoke overload ambiguities (had to consistently use raw pointer-typed arguments - `PCWSTR`/`PWSTR`/`&pi` instead of `out pi`/`string` - to force the compiler onto the raw extern overloads rather than the managed "friendly" overloads CsWin32 also generates).
- [x] **Step 10 — Real verification run (FR-2)**: SUCCESS. Ran against `_dest/usr/bin/msys-2.0.dll` (Unit 1a's vanilla build) with `--expect failing`. Created the real `AIDLC.AppContainerHarness` AppContainer profile on this machine for the first time (SID `S-1-15-2-1005668658-2595332598-2177366858-1034182510-3523517096-3749997732-888147124`). The `startup` scenario PASSED (i.e., correctly matched the expected failure signature): captured stderr was `0 [main] bash (34676) E:\Temp\appcontainer-harness\bin\bash.exe: *** fatal error - NtCreateDirectoryObject(\BaseNamedObjects\msys-2.0S5-611d995b890bea2a): 0xC0000022`, exit code `-1073741502` (0xC0000022 as signed int32) - an exact, independent reproduction of the bug described in `aidlc-docs/brief.md`. The other 6 scenarios reported "not applicable in failing mode" per BR-1 (by design), so `OVERALL: FAIL` for the suite is the *expected*, correct result in failing mode - the harness's own exit code was 1, which is correct behavior, not a bug.
