- [x] **Step 4 ‚Äî Modify `winsup/cygwin/wincap.cc`/`.h`**: Added `has_appcontainer_support` capability flag (`true` in all 10 existing OS-version tables, since AppContainer has existed since Windows 8).

Final build: clean, 0 errors.
- [x] **Step 9 ‚Äî Install build output**: `make DESTDIR="$(pwd)/_dest" install` succeeded; `_dest/usr/bin/msys-2.0.dll` produced.
- [x] **Step 10 ‚Äî Validate against the AppContainer harness**: **UPDATED, see patch-summary.md Round 5 ‚Äî the shared-namespace bug is now FIXED and confirmed.** After further investigation (Rounds 2-5, driven substantially by user pushback on premature "kernel wall" conclusions), root cause was found: the patch's original component-by-component `NtOpenDirectoryObject` walk failed on `\Sessions` (whose DACL grants nothing to AppContainer SIDs), while a **single** absolute-path open of the full `\Sessions\<n>\AppContainerNamedObjects\<sid>` string succeeds (confirmed empirically, matching how Win32's `CreateEventW` resolves AppContainer named objects internally). Rewrote `appcontainer_resolve_shared_parent_dir()` accordingly (`create_dir_path_relative` replaced with `open_appcontainer_object_dir`, one single-call open). Rebuilt and re-validated: `NtCreateDirectoryObject`/`STATUS_ACCESS_DENIED` on the shared namespace **no longer occurs at all** ‚Äî all 7 harness scenarios now get past this entire code path for the first time. A **new, different, later wall** was found further into startup (signal-pipe creation, `sigproc.cc`/`fhandler/pipe.cc` ‚Äî a separate subsystem) ‚Äî see patch-summary.md Round 5 for full detail and next steps.
- [x] **Step 11 ‚Äî Validate the non-AppContainer path still works (NFR-3)**: PASSED ‚Äî full smoke check (uname, echo, subshell fork, file I/O, env vars, pipe, exit code) all succeeded against the patched DLL in an ordinary unsandboxed shell; toolchain restored to its original state afterward.
