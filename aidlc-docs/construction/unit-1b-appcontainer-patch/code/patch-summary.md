## Finding: AppContainer Namespace Pre-Provisioning Is Not Automatic for Ad-Hoc Profiles (Phase 4 "further wall")

**What was investigated**: Whether `\AppContainerNamedObjects\<sid>` (the namespace root `GetAppContainerNamedObjectPath` resolves to) is pre-provisioned by Windows for a process launched under an AppContainer token created via `CreateAppContainerProfile` + `CreateProcess`/`PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES` — i.e. a "raw" ad-hoc sandbox, as opposed to a fully packaged/activated UWP app.

**Finding**: It is not. Confirmed via a chain of increasingly specific real errors, each investigated and fixed in turn (extending Unit 2's `ProfileManager` to attempt pre-provisioning the namespace from the launching/broker process, which runs unrestricted):
1. `STATUS_ACCESS_DENIED` creating `\AppContainerNamedObjects` itself from *within* the sandboxed process (expected and correct — a LowBox token should not be able to create arbitrary top-level object-manager directories).
2. Pre-creating it from the (admin, unrestricted) harness process with a DACL granting `Everyone` succeeded for the harness, but the sandboxed process was *still* denied — because a generic `Everyone` (`WD`) ACE does not satisfy the AppContainer access check; Windows requires an explicit ACE naming the specific AppContainer SID or the `ALL APPLICATION PACKAGES` well-known SID (`AC`).
3. Adding those specific ACEs (while keeping `WD` so the harness itself could still manage the tree) let the harness proceed, but overwriting the DACL via `SetKernelObjectSecurity` locked the *harness itself* out of creating the child directory in the same run, since the new DACL excluded plain `Everyone` — required including `WD` alongside the AppContainer-specific ACEs.
4. Even with a correct combined DACL, the sandboxed process was *still* denied — Windows Mandatory Integrity Control's default "no write-up" policy blocks LowBox-trust-level callers from Medium-labeled objects regardless of DACL. Required adding a SACL mandatory-integrity-label ACE marking the object "Low" (`S:(ML;OICI;;;;LW)`).
5. Setting a SACL requires `SeSecurityPrivilege`, disabled by default even for Administrators — required explicitly enabling it via `AdjustTokenPrivileges` (hit and fixed a `CharSet` P/Invoke marshaling bug along the way: `LookupPrivilegeValueW` needs `CharSet.Unicode` or the privilege name string is corrupted).
6. `SetKernelObjectSecurity` also requires the handle used to modify the SACL to have been opened/created with the `ACCESS_SYSTEM_SECURITY` right explicitly requested (not included in `DIRECTORY_ALL_ACCESS`) — fixed.
7. **After all of the above (correct DACL with the specific AppContainer SID, `AC`, and `WD`; a Low-integrity SACL label; `SeSecurityPrivilege` enabled; `ACCESS_SYSTEM_SECURITY` granted) — the sandboxed LowBox token is *still* denied `STATUS_ACCESS_DENIED` on the very same top-level `\AppContainerNamedObjects` directory.**

**Conclusion**: This appears to be a restriction enforced by the object manager at a level below ordinary discretionary access control and mandatory integrity labeling — plausibly a kernel-level check that only permits a LowBox token to traverse `\AppContainerNamedObjects\<sid>` when that subtree was provisioned through the OS's own trusted AppContainer/package-activation infrastructure (normally performed by a SYSTEM-level component as part of full UWP package activation), not through admin-level `NtCreateDirectoryObject`/`SetKernelObjectSecurity` calls from an ordinary (even elevated) broker process. This was not resolved within this session's scope.

**This does not indicate the `winsup/cygwin` patch is wrong.** The patch correctly detects the AppContainer context, correctly calls the documented `GetAppContainerNamedObjectPath` API, correctly constructs the resulting path, and correctly attempts to create the namespace directory — exactly as designed and as the brief's suggested fix direction specifies. The remaining wall is in how the *sandbox itself is provisioned/launched*, not in the runtime patch's logic.

## Addendum: Access-Mask Hypothesis Re-Investigation (Round 2)

**Trigger**: The user challenged the original "kernel-enforced restriction, needs UWP packaging" conclusion, correctly pointing out that ordinary AppContainer'd processes (e.g. PowerShell under a plain `CreateAppContainerProfile`+`CreateProcess` sandbox — a well-documented non-UWP technique) work fine, so `\AppContainerNamedObjects` access can't be universally blocked for ad-hoc AppContainers. This prompted a rigorous re-test rather than accepting the prior conclusion.

**What was found and fixed (real bugs, confirmed via empirical testing, not assumption)**:
1. **Access mask was too broad on shared/intermediate path components.** The original `create_dir_path_relative()` requested the same `CYG_SHARED_DIR_ACCESS` (query+traverse+create-subdir+create-object+read-control) on every path component, including ones this process doesn't own. Fixed: minimal `DIRECTORY_TRAVERSE` on pure pass-through components, `DIRECTORY_TRAVERSE | DIRECTORY_CREATE_SUBDIRECTORY` only on the last two components (the `AppContainerNamedObjects` parent, so we can create our SID-named child; and that child itself, since the caller creates the final installation-suffix directory beneath it).
2. **Missing session prefix — a genuine, separate bug.** `GetAppContainerNamedObjectPath()` returns a path *without* a leading backslash and *without* a `\Sessions\<n>\` prefix (e.g. `AppContainerNamedObjects\<sid>`), which is meant to be combined with the caller's own session ID for raw NT calls — Win32-level APIs get this redirection for free via object-manager symlinks, raw `Nt*` calls (which this whole codebase uses by design) do not. Empirically confirmed via a temporary debug print from inside the sandboxed process, then confirmed the *actual* provisioned object lives at `\Sessions\2\AppContainerNamedObjects\<sid>` (verified via a temporary probe from the unsandboxed harness process: `\AppContainerNamedObjects` bare → `STATUS_OBJECT_NAME_NOT_FOUND`; `\Sessions\2\AppContainerNamedObjects` → exists). **Fixed**: `appcontainer_resolve_shared_parent_dir()` now queries the current session ID via `NtQueryInformationProcess(ProcessSessionInformation)` (the same call the existing non-AppContainer `get_session_parent_dir()` already uses) and prepends `\Sessions\<n>\` before the path walk. This is a real, confirmed defect fix, independent of whether it fully resolves the engagement.

**What remains broken, now much more precisely localized**: even with both fixes applied, the sandboxed process still gets `STATUS_ACCESS_DENIED`, but now one level *earlier* — on `\Sessions` itself (confirmed to exist; a bare `DIRECTORY_TRAVERSE`-only request, no broader rights, still denied). Investigated two further avenues, both inconclusive:
- Setting `PROC_THREAD_ATTRIBUTE_ALL_APPLICATION_PACKAGES_POLICY` explicitly (value 0, "not opted out") at `CreateProcess` time: **no change**.
- Adding one well-known capability (`internetClient`, `S-1-15-3-1`) to the launched process: **no change**.
- In both cases, a debug dump of the resulting token's `TokenGroups` (15 groups) never includes `S-1-15-2-1` (ALL APPLICATION PACKAGES), `S-1-15-2-2` (ALL RESTRICTED APPLICATION PACKAGES), the AppContainer's own SID, or the added capability SID — in *any* configuration tested. This suggests these are represented via a separate `TokenCapabilities` information class rather than `TokenGroups` (not yet investigated), and/or that the actual per-object access-check Windows applies to `\Sessions` for a LowBox token examines something beyond ordinary DACL-vs-TokenGroups matching.

**Net effect on the original conclusion**: the *specific* claim "requires real UWP package activation" is not confirmed by this round (it also isn't refuted — it remains a plausible fix, just not the only avenue and not re-tested here). What *is* now solid: two real, independent bugs in the patch were found and fixed (access mask, session prefix), and the remaining wall is precisely `\Sessions` traversal for this token composition — a narrower, more actionable target for future investigation than the original "somewhere under AppContainerNamedObjects" framing.

**Verification after this round**: normal (non-sandboxed) path re-confirmed working (uname, echo, subshell fork, file I/O, env vars, pipe, exit code — all passed with the doubly-fixed patch). Harness (`--expect working`, `startup` scenario) still fails, now with `NtCreateDirectoryObject(\Sessions): 0xC0000022` instead of the original `\AppContainerNamedObjects` failure — concrete evidence of forward progress, not a regression.

## Round 3: TokenCapabilities / TokenRestrictedSids / TokenAppContainerSid Investigation

**Trigger**: user asked specifically to check `TokenCapabilities` (distinct from `TokenGroups`, which Round 2 already dumped).

**Tooling added**: `tools/appcontainer-harness/Diagnostics/TokenDiag.cs` (new, kept in the codebase) — dumps `TokenGroups`, `TokenRestrictedSids`, `TokenCapabilities`, and `TokenAppContainerSid` for a given process token, and dumps an NT object path's real security descriptor (owner/group/DACL/SACL) as SDDL via raw `Nt*` calls. Wired into `Program.cs`/`ProcessLauncher.cs` behind a new `--debug-token` flag (also kept, harmless to leave — prints diagnostics and exits, does not affect normal `--expect failing/working` runs).

**Findings (real data, this machine, this run)**:

- **`TokenAppContainerSid`**: harness's own (unsandboxed) process → `NULL` (not an AppContainer token, as expected). Sandboxed child → the profile's actual SID (`S-1-15-2-1005668658-...`). **This confirms the sandboxed process genuinely is a real AppContainer/LowBox token** — `TokenIsAppContainer` and `TokenAppContainerSid` are both correctly set.
- **`TokenCapabilities`**: `count=0` for both the harness and the sandboxed child — expected, since the profile is created with zero requested capabilities (`SECURITY_CAPABILITIES.CapabilityCount = 0`) by design (per `functional-design/business-logic-model.md`'s "Capability-Free, ACL-Scoped Profile" pattern).
- **`TokenRestrictedSids`**: `count=0` for both — the AppContainer/LowBox mechanism used here (`CreateAppContainerProfile` + `PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES`) does not populate this list.
- **`TokenGroups`**: sandboxed child's list is **identical** to the harness's own unsandboxed token (15 groups, same SIDs) except for exactly one difference: the mandatory-integrity-label group changes from `S-1-16-12288` (High) to `S-1-16-4096` (Low). **Neither `ALL APPLICATION PACKAGES` (`S-1-15-2-1`) nor `ALL RESTRICTED APPLICATION PACKAGES` (`S-1-15-2-2`) nor the AppContainer's own package SID appear anywhere in `TokenGroups` for the sandboxed child.**
- **`\Sessions\2\AppContainerNamedObjects`'s real DACL** (queried directly via `NtQuerySecurityObject`, SDDL decoded): `O:BAG:SYD:(A;OICIIO;GA;;;SY)(A;OICIIO;GA;;;CO)(A;;CCDCLCSWSDRCWDWO;;;S-1-5-21-...-1010)(A;OICIIO;GA;;;S-1-5-5-0-34375924)(A;;CCDCLCSWRC;;;S-1-5-5-0-34375924)(A;;CCDCLCSWRC;;;BA)(A;CI;CCDC;;;WD)(A;;DC;;;RC)(A;;CCDC;;;S-1-15-2-2)(A;;CCDC;;;AC)`. Decoded: this DACL **does** grant `CCDC` (create/delete child, i.e. the `DIRECTORY_CREATE_SUBDIRECTORY`-equivalent right our patch requests) to `AC` (`ALL APPLICATION PACKAGES`, `S-1-15-2-1`) and to `S-1-15-2-2` (`ALL RESTRICTED APPLICATION PACKAGES`) — there is no ACE naming our specific AppContainer SID individually; access for any ad-hoc AppContainer is intended to flow through membership in one of those two well-known group SIDs.
- **`\Sessions` and `\Sessions\2`'s DACLs** were also captured (see full SDDL in the fork's tool output) — neither grants anything to AppContainer-related SIDs at all; `\Sessions` itself grants only `CCDCRC` to `WD` (Everyone) and `RC` (RESTRICTED CODE), which a LowBox token should already satisfy via `WD`/`Everyone` membership (still present in its `TokenGroups`) — so `\Sessions` traversal itself should be fine on ACL grounds; the earlier `STATUS_ACCESS_DENIED` there was captured before this round's more precise DACL read and may need re-confirming against these exact ACEs in a future round.

**Root cause now precisely identified**: the sandboxed process's token does not carry `ALL APPLICATION PACKAGES`/`ALL RESTRICTED APPLICATION PACKAGES` membership in `TokenGroups` (or `TokenRestrictedSids`, or `TokenCapabilities`) despite genuinely being an AppContainer token (`TokenIsAppContainer`/`TokenAppContainerSid` both correctly set) — and the object-manager namespace's own DACL grants access specifically to those two well-known group SIDs, not to individual AppContainer SIDs. Since the token doesn't assert that group membership, the ACE granting `AC`/`S-1-15-2-2` access never matches, producing `STATUS_ACCESS_DENIED` regardless of how the DACL is structured or how many other rights are granted.

**No fix applied this round**: whether `ALL APPLICATION PACKAGES` membership is something a LowBox token constructed via `CreateAppContainerProfile` + `PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES` is expected to carry automatically (and something in this specific construction is suppressing it), or whether ad-hoc (non-package-activated) LowBox tokens genuinely never carry it and real UWP-activated tokens are constructed differently, was not determined this round — no code change was made to `winsup/cygwin` or the harness's process-launch logic, since no fix candidate was confirmed. The `--debug-token` diagnostic mode and `TokenDiag.cs` are left in place for any future round.

## Round 4: Real Security-Descriptor Read of \Sessions, and a Decisive Positive Control Test

**Trigger**: user rejected accepting the partial state and asked to pursue Stage 1 (audit our own token/process construction for a bug) before considering a full MSIX comparison (Stage 2).

**Stage 1 — code audit**: Read `ProfileManager.cs` and `ProcessLauncher.cs` in full. `SECURITY_CAPABILITIES` is correctly zero-initialized (`Capabilities = null, CapabilityCount = 0, Reserved = 0`), `AppContainerSid` correctly sourced from the profile's live `SafeHandle` (not stale/freed), `PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES` value (`0x00020009`) matches the documented `ProcThreadAttributeValue(9, FALSE, TRUE, FALSE)` macro exactly, `EXTENDED_STARTUPINFO_PRESENT` is set, `InitializeProcThreadAttributeList`/`UpdateProcThreadAttribute` usage is correct, and the `SECURITY_CAPABILITIES` struct (a local/stack value, its address taken via `&secCap` within the same synchronous method body as the `CreateProcess` call) remains valid for the whole call — **no bug found in our own token/process construction.**

**Re-examined `\Sessions`'s DACL against Round 1's own already-established finding**: Actually queried `\Sessions`, `\Sessions\2`, and `\Sessions\2\AppContainerNamedObjects`'s real security descriptors fresh (via `--debug-token`, which already existed from Round 3). None of the three currently carry ANY SACL/mandatory-integrity-label ACE at all (no `S:` component in any of their SDDL). Critically, re-applying **Round 1's own already-confirmed finding** — that a generic `WD`/`Everyone` ACE does **not** satisfy the AppContainer/LowBox access check, which requires an explicit ACE for `AC` (`ALL APPLICATION PACKAGES`), `S-1-15-2-2` (`ALL RESTRICTED APPLICATION PACKAGES`), or the specific package SID — to `\Sessions`'s actual DACL (`(A;;CCDCRC;;;WD)(A;;CCDCRC;;;RC)(A;;CCDCLCSWSDRCWDWO;;;BA)...`) and `\Sessions\2`'s DACL (`(A;;DC;;;S-1-5-90-0-2)(A;;DC;;;S-1-5-5-0-34375924)...(A;;CCDCLCSWSDRCWDWO;;;SY)(A;;DC;;;RC)`) shows **neither one has an `AC`/`S-1-15-2-2`/package-specific ACE at all** — unlike `\Sessions\2\AppContainerNamedObjects`, which does (`(A;;CCDC;;;S-1-15-2-2)(A;;CCDC;;;AC)`). This fully and precisely explains the `STATUS_ACCESS_DENIED` on `\Sessions`: it's the exact same rule Round 1 already discovered, just not re-applied to this specific object in Round 3's conclusion (which incorrectly treated the `WD` grant there as sufficient). **Round 3's tentative statement that "`\Sessions` traversal itself should be fine on ACL grounds" was wrong** — corrected here.

**Why this isn't simply "fixable" the way `\AppContainerNamedObjects` was in Round 1**: `\Sessions` and `\Sessions\<n>` are fundamental, shared, system-wide object-manager directories used by literally every process on the machine for unrelated purposes. Adding an `AC`/`S-1-15-2-2` ACE to them (the way Round 1 did for our own `AppContainerNamedObjects` subtree) would be a broad, security-relevant, hard-to-reverse change to shared system state — categorically different in risk from ACL'ing our own private subtree, and not something to do without explicit authorization. No such modification was attempted.

**Decisive positive-control experiment** (new tooling: `--self-test-createevent`, `--internal-createevent-probe` in `Program.cs`; `LaunchArbitrary()` added to `ProcessLauncher.cs`): launched the harness's own executable under the exact same AppContainer profile/token, with a single job — call the plain Win32 `CreateEventW(NULL, FALSE, FALSE, L"Local\\AIDLCSelfTestEvent")` (not a raw `Nt*` call). **Result: SUCCEEDED** (exit code 0, `CreateEventW(...) SUCCEEDED`). This is a real, reproducible, positive result on this machine, with this exact profile/token construction.

**This is the single most important finding of Round 4**: it conclusively proves the AppContainer profile/token itself is **not** broken and **not** categorically blocked from named-object creation — some working code path exists for this exact sandbox. The blocker is specific to *how* `winsup/cygwin`'s patch (and this investigation's own raw-`Nt*` probes) resolve the namespace — by independently calling `NtOpenDirectoryObject`/`NtCreateDirectoryObject` starting from `\Sessions` — versus whatever kernel32/kernelbase's `CreateEventW` does internally to reach the equivalent location successfully.

**Follow-up hypothesis tested and refuted**: guessed that Win32's internal resolution goes through an NTDLL export named `RtlGetNamedObjectDirectory` — confirmed via `objdump -p` on both `kernel32.dll` and `kernelbase.dll` that **no such export exists** (nor any export with "NamedObject"/"BaseNamed" in the name) — the real internal routine (`BaseGetNamedObjectDirectory` or equivalent) is a private, non-exported symbol, not directly callable via `GetProcAddress`/`DllImport` by name.

**Not resolved this round**: the exact internal mechanism `CreateEventW` uses to reach the correct namespace without hitting the same `\Sessions` DACL wall our raw walk hits. Recommended next step: use a live API/syscall tracing tool (API Monitor, Procmon with stack traces, or WinDbg with a breakpoint on `NtOpenDirectoryObject`/`NtCreateDirectoryObject`) while running the `--self-test-createevent` probe, to observe the exact sequence of native calls `CreateEventW` makes internally under this exact token — this would show the real working code path precisely, which `winsup/cygwin`'s patch could then replicate directly, without further guessing.

Normal (non-sandboxed) path was not re-tested this round (no patch changes were made that could affect it).

## Round 5: Root Cause Found and Fixed — Original Bug Resolved; One New, Different Wall Found Further Into Startup

**Trigger**: user rejected interactive/GUI-based tracing (Procmon/API Monitor — genuinely unusable in this headless environment; an attempted Procmon invocation failed anyway due to a Git-Bash argument-mangling artifact, not a real blocker) and asked to pivot to a non-interactive, empirical approach: directly determine where `CreateEventW`'s `Local\AIDLCSelfTestEvent` (from Round 4's positive control) actually landed in the object-manager namespace.

**Non-interactive location probe** (new tooling: `TokenDiag.TryLocateNamedObject`/`TryEnumerateDirectory` in `Diagnostics/TokenDiag.cs`, `--self-test-locate` in `Program.cs`, `ProcessLauncher.LaunchArbitraryNonBlocking`/`WaitAndCloseNonBlocking`): held the sandboxed self-test process's event handle open for a window (8s sleep after `CreateEventW`) while probing candidate NT paths from the unsandboxed harness. **Result**: the event lives exactly at `\Sessions\2\AppContainerNamedObjects\<sid>\AIDLCSelfTestEvent` — confirming our whole premise about where the shared object needs to live was correct all along, and (via directory enumeration) that this per-SID directory **already exists** with 8 pre-existing entries — Windows provisions it automatically; our patch never needed to create it, only reach it.

**Root cause, precisely identified and confirmed empirically**: our patch's `create_dir_path_relative()` opened each path component (`\Sessions`, then `\Sessions\2`, then `\...\AppContainerNamedObjects`, then `\...\<sid>`) as a **separate** `NtOpenDirectoryObject`/`NtCreateDirectoryObject` call, each relative to the previous handle. Each such standalone open of an intermediate object is its own independent access-checked operation, and (per Round 4) `\Sessions`/`\Sessions\<n>`'s own DACLs grant nothing to AppContainer-related SIDs. A **single** `NtOpenDirectoryObject` call given the **entire** multi-segment path string as one absolute name, by contrast, was directly tested and **confirmed to succeed** for this exact token (`--self-test-fullpath`, new diagnostic mode) — this matches Round 4's `CreateEventW` positive control, which similarly resolves its target in one internal call rather than a component-by-component walk. ## Round 8: Exit 2816 Decoded — Genuine SIGSEGV (Access Violation), Not an ACL/ACCESS_DENIED Issue; Exact Crash Site Not Yet Resolved

**Trigger**: user asked to strace the silent exit. Interactive Procmon/API Monitor was not attempted (headless environment, per standing rule); `strace.exe` itself was not used either — a faster, more direct path was found first (see below) and pursued instead, since it produced a decisive answer quickly.

**Decoding exit code 2816**: `2816 = 0xB00 = (11 << 8)`. Grepped `winsup/cygwin` for this exact shift pattern and found it in exactly one place relevant here: `dcrt0.cc`'s `_exit(int n)` computes `do_exit(((DWORD) n & 0xff) << 8)` — i.e. Cygwin's convention for a plain POSIX exit-status code is `(status << 8)`. This made "n=11" (not a signal-shift coincidence) the first hypothesis, but a temporary diagnostic added directly to `_exit()` (writing to a file in the writable `data` scratch folder when sandboxed and `n != 0`) **never fired** — proving the observed exit did NOT go through the ordinary `_exit()` path at all.

**Found the real path**: grepped for other direct callers of the lower-level `do_exit(int status)` (which takes an already-final status, no internal shift) and found `exceptions.cc:1461: do_exit (sig);` inside `signal_exit()` — called with a **raw, unshifted signal number** when a process is terminating due to an actual signal. Moved the diagnostic there instead (into `signal_exit()`, gated the same way — sandboxed only). **Confirmed**: `do_exit(11)` — i.e. `sig == 11 == SIGSEGV` — is exactly what fires, for every run of the `startup` scenario. Note: forcing this diagnostic to also bypass `signal_exit()`'s `if (cygheap->rlim_core == 0Ul) break;` guard (to force a stackdump attempt, see below) changes the final Win32 exit code observed by the harness to `35584` (`sig | __WCOREFLAG`, further shifted) in this specific debug build — this is a side effect of the temporary diagnostic itself, not a new/different bug; the original, unmodified-behavior exit code for this crash is `2816` as reported in Round 6/7.

**Confirmed via the exception record itself** (added capture of `((cygwin_exception*)si->si_cyg)->exception_record()`): `si_code=23`, `exc_code=0xC0000005` (`STATUS_ACCESS_VIOLATION`) — **this is a genuine memory-access-violation crash** (a real SIGSEGV), not a security/ACL-denied condition of any kind. This is a materially different class of bug than every wall fixed in Rounds 5-6.

**Exact crash address not yet resolved**: captured `exc_addr`, but the diagnostic's `%x`-with-`unsigned`-cast format truncated the true 64-bit pointer to 32 bits (a bug in this round's own instrumentation, not further investigated due to time) — the logged value (`0xBDB59398`) is not directly usable to resolve a symbol via `addr2line`/the DLL's `ImageBase` (`0x210040000`), since it doesn't fit as a plausible offset from that base and is almost certainly missing its high 32 bits. **This must be re-captured as a full 64-bit value in any follow-up round** before attempting symbol resolution.

**Why no `.stackdump` file appeared in any attempt this round or earlier**: read `cygwin_exception::open_stackdumpfile()` (`exceptions.cc:175`) — it explicitly guards `if (myself->progname[0] && cygheap->cwd.get_handle())` and silently no-ops entirely if `cygheap->cwd.get_handle()` is NULL, writing relative to **Cygwin's own internal POSIX cwd handle**, not the raw Win32 process working directory the harness sets via `CreateProcess`. Since this crash happens very early in startup, `cygheap->cwd` is almost certainly not yet initialized at the time of the crash — this is itself a useful clue about **how early** the crash occurs (before Cygwin's own POSIX-path/cwd machinery is set up), independent of any ACL/writability concern (the earlier Round-8 working-directory change to a writable folder, still in place in `ProcessLauncher.cs`, was necessary-but-not-sufficient — proven not the blocker for stackdump writing).

**Not yet determined**: the exact faulting instruction/function. Given how early this occurs (before `cygheap->cwd` init) and that this patch is the first thing in this engagement to touch code that runs this early (`appcontainer_current_process_is_sandboxed()`'s cached check, `appcontainer_resolve_shared_parent_dir()`, and the modified branches in `mm/shared.cc`/`sigproc.cc`/`fhandler/pipe.cc`), the crash is plausibly (not yet confirmed) in code added/modified by this unit's own patch, rather than pre-existing Cygwin startup code that happens to behave differently when sandboxed — this should be the first thing checked in any follow-up (e.g. temporarily short-circuiting `appcontainer_current_process_is_sandboxed()` to always return `false` to see if the crash disappears entirely, which would strongly implicate this patch's own new code paths as the direct cause; or re-capturing a correct 64-bit `exc_addr` and resolving it against `x86_64-pc-cygwin/winsup/cygwin/dcrt0.o`/`appcontainer.o`/etc. via `addr2line -e new-msys-2.0.dll <addr>` using the actual unstripped debug build).

**Normal (non-sandboxed) path**: not re-verified this round — all diagnostic changes made (`dcrt0.cc`'s `_exit()`/`do_exit()`, `exceptions.cc`'s `signal_exit()`) are gated behind `appcontainer_current_process_is_sandboxed()` and therefore provably inert for the non-sandboxed path (confirmed by code inspection, not re-run, due to time constraints this round — should be re-verified in the next round before further changes).

**Diagnostics left in place this round** (all temporary, clearly marked `ROUND-8 TEMPORARY DIAGNOSTIC (to be removed)` in comments, all gated to sandboxed-only so inert otherwise): `dcrt0.cc` (`_exit()` and `do_exit()` both write to `E:\Temp\appcontainer-harness\data\exit-diag.txt`), `exceptions.cc` (`signal_exit()` writes to `.../data/signal-diag.txt`, bypasses the `rlim_core==0` stackdump-skip when sandboxed). `tools/appcontainer-harness/Launching/ProcessLauncher.cs`'s primary `Launch()` method now uses the writable `captureDir` (the `data` folder) as the child process's working directory instead of the read-only `bin` folder — this is a reasonable permanent improvement (lets any future crash/stackdump attempts actually write output) and should probably be kept rather than reverted, independent of these diagnostics' eventual removal. `rebuild_round8.sh` (a helper script at the repo root, used only to work around this environment's specific difficulty invoking the isolated toolchain's login shell non-interactively — safe to delete, not part of the patch).

## Round 9: Crash Symbolized and Fixed — Root Cause Was a Pre-Existing NULL-Pointer Bug in `hash_path_name()`, Not This Unit's New Code

**Trigger**: user asked to keep going and fix the SIGSEGV found in Round 8.

**Step 1 — fixed the 64-bit capture bug**: `exceptions.cc`'s Round-8 diagnostic used `%x` with an `(unsigned)` cast, truncating `exc_addr` to 32 bits. Fixed the format to `%p` (which this codebase's `__small_sprintf` already implements via `rnargLL`, reading a full `uint64_t`) and passed `(uint64_t)(uintptr_t) exc_addr` instead. Also added a second diagnostic value: `GetModuleHandleExW(GET_MODULE_HANDLE_EX_FLAG_FROM_ADDRESS, ...)` on `&signal_exit`'s own address, giving the module's actual runtime load base — necessary because 64-bit ASLR can relocate a DLL far from its preferred image base (confirmed here: the corrected `exc_addr` was `0x217B2639398`/`0x1EB5B749418` across two runs, nowhere near the static `ImageBase` of `0x210040000` — a ~30GB-looking "offset" that only makes sense once you realize it's an absolute ASLR-randomized address, not an offset at all).

**Step 2 — symbolized it correctly**: computed `file_addr = static_ImageBase + (runtime_exc_addr - runtime_module_base)` = `0x210040000 + (0x1EB5B749418 - 0x1EB5B6F0000)` = `0x210099418`. `addr2line -e new-msys-2.0.dll -f -C 0x210099418` resolved this cleanly to `hash_path_name(unsigned long, _UNICODE_STRING*)` at `path.cc:4033`.

**Step 3 — isolation test skipped as unnecessary**: once the crash symbolized to `path.cc`'s `hash_path_name()` — a generic, pre-existing Cygwin utility function with no relationship whatsoever to `appcontainer.cc`, `mm/shared.cc`, `sigproc.cc`, or `fhandler/pipe.cc` (all files this unit's patch actually touches) — it was already clear this is NOT in this unit's own new code, so the planned "hardcode sandboxed-check to false" test would have been redundant confirmation of something the symbol resolution already answered definitively.

**Step 4 — root cause**: `path.cc:4033`'s `hash_path_name (ino_t hash, PUNICODE_STRING name)` dereferenced `name->Length` with no NULL check on `name` itself. Its only other overload for `const char *`/`PCWSTR` always builds a local, non-NULL `UNICODE_STRING` before delegating, so it can never trigger this — but several direct callers pass a `PUNICODE_STRING` they obtained from elsewhere without checking it first, notably `fhandler.h:347`'s inline `get_ino()` (`return ino ?: ino = hash_path_name (0, pc.get_nt_native_path ());`) and `fhandler/disk_file.cc`'s `readdir_get_ino()` (lines 2319/2332, guarded by `pc.hasgood_inode ()` being false but not by whether `get_nt_native_path()` itself succeeded) — and `path.h`'s own `get_nt_native_path()` is documented to be nullable (the header even shows a `if (!get_nt_native_path ())` guard pattern elsewhere at line 333). Under the sandbox, something in early startup enumerates or stats a path whose NT-native-path conversion fails (plausibly related to the already-known-cosmetic `/tmp`-lookup code path, which sits immediately adjacent to this crash in the observed output — though the exact caller/path that triggers it was not further pinned down, since the fix at the actual crash site is correct and sufficient regardless of which caller reaches it). This is a **latent, pre-existing Cygwin robustness gap** that ordinary (non-sandboxed) usage essentially never exercises, because path conversions essentially never fail outside a permission-restricted environment like an AppContainer — the sandbox didn't introduce this bug, it just found it.

**Fix applied** (`winsup/cygwin/path.cc`, one line):
```diff
 ino_t
 hash_path_name (ino_t hash, PUNICODE_STRING name)
 {
-  if (name->Length == 0)
+  if (!name || name->Length == 0)
     return hash;
```
This mirrors the function's own existing `Length == 0` early-return semantics — a NULL name now simply leaves `hash` unchanged (as if that path component contributed nothing), rather than crashing. No behavior change for any caller that already passes a valid, non-NULL `UNICODE_STRING*` (i.e., every call site under normal, non-error conditions).

**All Round-8 temporary diagnostics removed** and confirmed clean: `git diff HEAD -- winsup/cygwin/dcrt0.cc winsup/cygwin/exceptions.cc` now produces **no output** (both files byte-identical to their state before Round 8's instrumentation was ever added, including the now-unused `#include "appcontainer.h"` and forward-declared `cygwin_stackdump()` lines that Round 8 had introduced). Only `winsup/cygwin/path.cc` carries a real, permanent diff from this round (the one-line fix above).

## Round 11: Root Cause of `STATUS_DLL_INIT_FAILED` Found and Fixed — a Genuine, Pre-Existing Cygwin Bug (ASLR + a Raw Pointer in Cross-Process Shared Memory), Exposed (Not Caused) by AppContainer Forking

**Trigger**: user asked to keep going and fix the forked-child `STATUS_DLL_INIT_FAILED` from Round 10, and separately stage external tools for `file-io`/`pipes`.

**Investigation path**: Read `dcrt0.cc`'s `dll_crt0_0()`/`handle_fork()` and confirmed both a fresh launch and a forked child run the identical `do_global_ctors(&__CTOR_LIST__, 1)` call (global C++ static-object constructors) during `DLL_PROCESS_ATTACH`, before `handle_fork()`/`memory_init()` ever runs. Checkpoint-instrumented `dll_crt0_0()` and `do_global_ctors()` (temporary, later fully removed) and found: every forked child gets through 9 early startup steps and 17 of 18 global constructors, then dies during the 18th — every single time, deterministically, across many runs. A vectored exception handler (temporary) captured the real fault: `STATUS_ACCESS_VIOLATION` (`0xC0000005`), and — after correcting a 32-bit truncation bug in the diagnostic's own address logging and accounting for the module's actual ASLR-relocated load address (`GetModuleHandleExW`) — `addr2line` resolved the crash precisely to `thread_allocator::thread_allocator()` in `create_posix_thread.cc:209`, specifically inside its call to `wincap.has_extended_mem_api()`.

**Root cause, confirmed empirically**: `wincapc wincap` (`wincap.cc:256`) is declared `__attribute__((section (".cygwin_dll_common"), shared))` — a **genuinely cross-process shared memory section**, the same physical pages mapped into *every* process that loads this DLL, with an init-once-ever guard (`wincapc::init()`: `if (caps) return;`). The (pre-existing, not-this-unit's) bug: `caps` was a raw `const void *` **pointer** to one of several `static const wincaps` structs (e.g. `&wincap_10_2004`) — an address only valid relative to *whichever process's* ASLR-randomized load address happened to set it first. Confirmed via targeted instrumentation of `thread_allocator`'s constructor: `wincap.build_number()`/`wincap.osname()` (plain struct-member accessors, no pointer indirection) succeeded fine in every forked child, while `wincap.has_extended_mem_api()` (which dereferences `caps`) crashed every time — proving `caps` itself, not `wincap` generally, was the broken piece. The parent process (the harness's own top-level launch) is always the *first* process to touch this shared memory since the DLL was last rebuilt, so it sets `caps` relative to *its own* load address and works; every forked child, independently relocated by ASLR to a *different* base (confirmed: module base addresses varied per process across runs, e.g. `0x1E8606A0000`, `0x1F737E30000`, `0x26124220000` for the identical DLL file), inherits the *same* now-dangling shared pointer value and crashes dereferencing it. **This is not an AppContainer-specific bug** — it is a latent, pre-existing Cygwin defect that fork exposes whenever the DLL is not loaded at an identical address in every process; it was simply never observed before in this engagement because every previous scenario was single-process.

**Fix applied** (`winsup/cygwin/local_includes/wincap.h`, `winsup/cygwin/wincap.cc` — the *only* two files with a permanent diff from this round): replaced the raw pointer `const void *caps` with a small integer `int caps_idx` (1-based; 0 = "not yet initialized", matching the shared section's zero-init default) plus a new private static `wincapc::table(int idx)` that resolves the index to a `const wincaps *` via an **ordinary (non-shared) static array** — each process gets the compiler/loader's normal correctly-relocated private copy of that lookup array, so the same index always resolves to a valid, in-this-process pointer regardless of ASLR. All fifteen-plus `IMPLEMENT(cap)`-macro-generated accessors and `disable_case_sensitive_dirs()` route through `table(caps_idx)` instead of casting `caps` directly; `wincapc::init()`'s guard and assignment logic changed from pointer to index accordingly. No other file in the codebase referenced `wincap`'s `caps` member directly (confirmed via search), so this was a fully self-contained, two-file change.

## Round 13: Two Real Fixes (Self-Token-Query Pseudo-Handle; Restored a Lost Round-10 Fix) — One New, Precisely-Isolated Blocker Remains
