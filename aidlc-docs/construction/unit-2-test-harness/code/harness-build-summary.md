## Implementation Note: CsWin32 friendly vs. raw overloads
CsWin32 generates two overloads per Win32 API where possible: a "friendly" managed one (uses `string`, `out`, `SafeHandle`) and a raw `extern` one (uses `PCWSTR`/`PWSTR`, pointers, raw `HANDLE`). Mixing argument styles across a single call (e.g. `out pi` with otherwise-raw-pointer arguments) caused the compiler to fall back to matching against the friendly overload and report confusing type-mismatch errors against it. Fix: consistently use raw pointer-typed arguments (`PCWSTR`/`PWSTR` via `fixed`, `&pi` instead of `out pi`, explicit `(SECURITY_ATTRIBUTES*)null` casts) throughout `ProcessLauncher.cs` to force resolution onto the raw `extern` overloads, which is what's needed anyway for `STARTUPINFOEXW`/`PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES` support (the friendly `CreateProcess` overload only accepts a plain `STARTUPINFOW`, not the extended attribute-list form needed for AppContainer launch).

Also: `STARTUPINFOEXW` and `SECURITY_CAPABILITIES` are not auto-generated by CsWin32 just from listing the *functions* that use them (`CreateProcessW`, `UpdateProcThreadAttribute`) — they had to be listed explicitly as their own entries in `NativeMethods.txt`.
